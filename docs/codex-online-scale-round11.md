# Codex実装指示書: オンライン対戦を20人規模に耐える構造へ（v0.14.0）

## 目的

現状オンライン対戦は `rooms/{code}` という**単一ドキュメント**に全プレイヤーの状態を
`players` マップフィールドとして詰め込み、全員がそのドキュメント1つを購読している。
このため 20人規模で以下の問題がある:

- **読み取りのN²増幅**: 誰か1人の書き込み → 接続中の全員に読み取りが飛ぶ
- **単一ドキュメントへの書き込み集中**（Firestoreは1ドキュメント約1write/secが上限）

**見た目・操作・ゲーム内容は一切変えず**、裏側のデータ構造だけ作り替えて軽くする。

## 目標データモデル

プレイヤー状態を**サブコレクションに分割**する:

- `rooms/{code}` … ルーム共有状態のみ:
  `host, phase, round, funds, horseSeed, raceSeed, names, betDeadlineAt, resultDeadlineAt, gameOver`
  （このドキュメントは「フェーズ遷移」「シード配布」など**低頻度**の書き込みだけになる）
- `rooms/{code}/players/{uid}` … 各プレイヤー1ドキュメント:
  `name, balance, betDone, tickets, bankrupt, reviveResult, readyNext` など現在 players マップに入っている各項目

これにより、各プレイヤーのベット確定は**自分のサブドキュメントへの書き込み**になり、
単一ドキュメント集中が解消される。読み取りもプレイヤー単位の差分配信になり増幅が大幅に減る。

## 実装方針（重要: UIとゲームロジックは不変に保つ）

- **内部表現は現状維持**: 購読側で「ルームドキュメント」＋「playersサブコレクション」の2つのリスナーを張り、
  それらをマージして**従来と同じ `room.players` 形（{uid: {...}}）のオブジェクト**に組み立てて
  `onRoom(room)` 以降へ渡す。こうすれば `renderLobby` / `renderPlayerList` / `allBet` /
  `handleBetting` / `handleResult` などUI・判定側の変更を最小化できる。
- `subscribe()` を2リスナー化。どちらかが更新されたら合成して `onRoom` を呼ぶ。
  初回は両方そろってから描画する（プレイヤーリストが一瞬空になるのを防ぐ）。
- **書き込みの置き換え**（機能は同じ、宛先だけ変える）:
  - `createRoom`: ルームドキュメント作成＋自分のplayerサブドキュメント作成
  - `joinRoom`: ルーム存在確認後、自分のplayerサブドキュメントを作成/更新
  - ベット確定（`players.${uid}.betDone/tickets`）→ 自分のplayerサブドキュメントへ
  - `doLeave`: 自分のplayerサブドキュメントを削除。最後の1人なら
    playersサブコレクションを空にしてルームドキュメントも削除
  - ホスト精算（`hostSettle` 相当）: 各プレイヤーのbalance等を**各playerサブドキュメントに**書く。
    Firestore の `writeBatch` を使って1バッチでまとめて書くこと（書き込み回数を抑える）
  - `hostReset`: 同様に writeBatch で各playerサブドキュメントをリセット
  - ホスト交代ロジック（最若番uidが引き継ぐ）は players一覧から判定できるので維持
- **精算の権威はホストのまま**（今と同じ）。各クライアントが勝手に自分のbalanceを書くのではなく、
  ホストが全員ぶんを batch で確定する現状の設計を維持する。
- レース本体は今まで通り `raceSeed` から各端末で決定論再生（**ここは変更不要**）。

## Firestoreルール（重要・別途公開が必要）

Firestoreのルールはサブコレクションに**自動継承されない**。
`firestore.rules` を更新して `rooms/{code}/players/{uid}` を許可する match を追加すること
（既存の tournaments / rooms / meta / users / invites は消さない）。
※このファイルを更新しても本番反映はオーナーがFirebaseコンソールで手動公開する必要がある。
　実装後、reポート冒頭に「コンソールで新ルールの公開が必要」と明記すること。

## 検証（必須・自己申告禁止）

`scripts/` に **Node製の多人数負荷検証スクリプト**を新規作成すること（例 `verify-online-scale.mjs`）:

- Firebase JS SDK（既に使用中のCDN版でなく、`npm i --no-save firebase` でNode用に読み込む形でよい）で
  **20クライアントを模擬**し、1部屋に参加 → 全員ベット確定 → ホストがレース開始 → 精算 → 次ラウンド、を数ラウンド回す
- 計測して出力する: 総書き込み回数、総読み取り(スナップショット)回数、
  **1クライアントあたりの受信スナップショット数**、単一ドキュメントへの最大同時書き込み集中度
- 旧構造との比較になるよう、少なくとも「20人・3ラウンドで読み取りが N² 的に増えていないこと」を数値で示す
- テストは実プロジェクト（prospia-d5526）に書き込むことになるため、
  **専用の検証用ルームコード（例 `ZZ` で始まる）を使い、テスト終了後に必ず部屋を削除**してクリーンにすること

※このスクリプトは本番デプロイには含めない（scripts/配下でよい。node_modулesは.gitignore済み）。

## 制約（毎回同じ・必読）

- ビルドなし静的サイト維持、Vanilla JS + three.js (importmap)。本番のFirebase読み込みはCDN版のまま
- **見た目・操作・ゲームバランス・レース映像は一切変えない**（今回はネットワーク層だけ）
- オンラインのシード同期（同一シード→全端末同一結果）を壊さない
- ローカル対戦モードには影響を与えない
- 完了時 `src/version.js` を APP_VERSION 0.14.0 / APP_BUILD 19 に、index.htmlのバッジも更新
- タスクごとに日本語メッセージで独立コミット。push・デプロイはしない
