# Codex実装指示書: オンラインの読み取り回数をO(N²)→O(N)へ（v0.15.0）

## 背景（前ラウンドの実測で判明した事実）

前ラウンドでプレイヤー状態を `rooms/{code}/players/{uid}` サブコレクションへ分割し、
**書き込み集中は解消（maxConcurrentWrites=1）**できたが、
**Firestoreの読み取り回数は減らなかった（20人3ラウンドで配信3,800、旧推定1,380より多い）**。

原因は「N人が書く × N人が購読 = N²の配信」という扇形拡散が残っているため。
今回はこのN²を**O(N)に落とす**のが目的。書き込み集中の解消（各自が自分のドキュメントに書く）は維持する。

## 目標アーキテクチャ：ホスト集約 + 要約配信

**全員が全員を購読するのをやめる。**

- `rooms/{code}/players/{uid}` … 各プレイヤーが**自分の状態を自分のドキュメントに書く**（現状維持。書き込み集中しない）
- `rooms/{code}` ルームドキュメントに **`summary` フィールド**を持たせる。
  これは全クライアントの描画に必要な最小情報だけを1つにまとめたもの:
  各プレイヤーの `{name, balance, betDone, bankrupt, reviveResult, rank?}` の配列/マップ。
  **この summary を書くのはホストだけ**（単一ライター＝書き込み集中なし。高頻度更新はデバウンスする）。
- **購読の張り方を役割で変える**:
  - **非ホスト**: `rooms/{code}` ルームドキュメント**1つだけ**を購読する。
    プレイヤー一覧・状態は `room.summary` から描画する（playersサブコレクションは購読しない）。
  - **ホスト**: ルームドキュメント + playersサブコレクションを購読する。
    playersに変化があったら（デバウンス: 例 250〜500ms）summaryを再計算して
    ルームドキュメントに1回書き込む。
- これにより、20人がベットしても**非ホストは「ホストのsummary更新」を受け取るだけ**になり、
  配信が N×N から「ホストのsummary更新回数 × N」に激減する。
  精算も、旧構造の「全員ぶん個別配信」ではなく「summary1回 → N配信」で済む。

## UI・ゲーム内容は不変に保つ（重要）

- `onRoom(room)` から先の描画・判定コードが期待する **`room.players`（{uid:{...}}）の形は維持**する。
  - 非ホスト: `room.summary` を `room.players` 形に展開して従来コードへ渡す。
  - ホスト: 従来どおり players 購読結果を `room.players` 形にして渡す（ホストは自分の集約元を持つ）。
- 見た目・操作・レース映像・ゲームバランスは一切変えない。今回もネットワーク層だけ。

## 設計上の必須事項

- **ホスト交代**: 現状の「最若番uidが引き継ぐ」を維持しつつ、
  新ホストは (1) playersサブコレクションの購読を開始し (2) summary更新を再開すること。
  非ホストはplayersを購読しないので、ホスト不在の短時間は summary が更新されないだけ（許容）。
  ただしホスト不在を検知したら速やかに引き継ぐこと（今の仕組みで可）。
- **summaryの書き込み頻度**: ホストがplayers変化のたびに即書きすると連打時に単一ドキュメントへ
  高頻度書き込みになる。**デバウンス**して最大でも数百msに1回に抑える。
  ただしフェーズ遷移（betting→race→result）に伴う重要な更新は取りこぼさないこと。
- **全員ベット完了の判定**はホストがplayers購読で行う（現状の allBet 相当）。
- ベット確定・参加・退出など**各プレイヤーの書き込みは自分のplayerドキュメントのまま**。
- レース再生（raceSeed 決定論同期）は変更不要。

## Firestoreルール

`rooms/{code}` と `rooms/{code}/players/{uid}` の許可は前ラウンドで追加済み。
summaryはルームドキュメントのフィールドなので**ルール変更は不要**（追加が必要なら firestore.rules を更新し、その旨を報告冒頭に明記）。

## 検証（必須・自己申告禁止）

前ラウンドの `scripts/verify-online-scale.mjs` を今回の設計に合わせて更新し、実プロジェクトで計測すること:

- 20クライアント×3ラウンド（1人がホスト役、19人が非ホスト役）
- **非ホストクライアントが受け取る配信ドキュメント/コールバック数**を計測して出力
- 受け入れ条件:
  - `maxConcurrentWritesToOneDocument` が小さいまま（summaryはホスト単独書き込み＝1）
  - **総配信数が前回（3,800）から大幅に減り、Nにほぼ比例**していること（N²的でない）を数値で示す
  - 非ホスト1人あたりの配信が、プレイヤー数を増やしてもほぼ一定〜緩やかにしか増えないこと
    （可能なら10人と20人で測って比較し、非ホスト当たり配信がほぼ横ばいなことを示す）
- 検証用ルーム（ZZ始まり）を使い、終了後に必ず削除
